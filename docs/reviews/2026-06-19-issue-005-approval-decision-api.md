## Database Query Proof

The artifact [2026-06-19-issue-005-api-proof.json](2026-06-19-issue-005-api-proof.json) is generated by a reproducible script that:

1. Starts a disposable `postgres:16` container.
2. Runs the repository migration runner.
3. Seeds a multi-user approval graph.
4. Builds and starts the Next.js app on port `3025` with `DATABASE_URL`, `HOOKWIRE_DATABASE_ROLE=hookwire_app`, and `HOOKWIRE_INTERNAL_API_SECRET`.
5. Calls the actual approve and deny HTTP endpoints with signed internal identity headers.
6. Queries `approval_requests`, `approval_decisions`, and `audit_events`.

The queried rows prove:

- Approved request status is persisted as `approved`.
- Denied request status is persisted as `denied`.
- Expired request status is persisted as `expired` without decision or audit rows.
- Unauthorized repeat probe remains `pending`.
- Decision rows record `user_id`, `source: web`, `decision`, `scope`, and `reason`.
- Audit rows record `actor_user_id`, `event_type`, `entity_id`, and metadata with source, actor, decision, scope, reason, route, risk, and decision id.

## Acceptance Criteria Mapping

- Only pending requests can be decided: API tests reject already-decided and expired requests; proof shows expired status without decision/audit rows.
- Decisions are safely rejected on repeat: API tests and proof show repeat approve and repeat deny return `409 already_decided` with one decision and one audit row.
- The deciding user is recorded: tests and proof assert `approval_decisions.user_id` and `audit_events.actor_user_id`.
- Request status changes are persisted: tests and proof query `approval_requests.status`.
- Audit event records source, actor, decision, scope, and reason: tests and proof assert audit metadata and actor columns.
- Decision validation against user eligibility: tests cover viewer and wrong-organization users returning `403 unauthorized`.
- Required reason and allowed scopes: tests cover missing denial reason and invalid scope validation.
- Transaction boundaries: service test injects a failure before audit insert and asserts request status, decision row, and audit row all roll back.

## Notes

- Eligibility is currently organization membership based: `owner`, `admin`, and `member` can decide; `viewer` and users outside the organization cannot. Route/group-specific eligibility can build on this in later integration and routing issues.
- Denial reason is required by this API. Approval reason is optional and persisted when supplied.
- The API requires a trusted internal identity signature. `x-hookwire-organization-id` and `x-hookwire-user-id` must be paired with `x-hookwire-identity-signature`, an HMAC over `organizationId:userId` using `HOOKWIRE_INTERNAL_API_SECRET`.
- The service sets `app.current_organization_id` inside the transaction and supports `HOOKWIRE_DATABASE_ROLE=hookwire_app`, so proof and tests exercise the schema's row-level tenant isolation contract rather than relying on superuser bypass.

Disposition:

- Added signed internal identity verification with `HOOKWIRE_INTERNAL_API_SECRET`.
- Added tests proving missing and invalid identity signatures are rejected before decision processing.
- Set `app.current_organization_id` at transaction start and added optional `HOOKWIRE_DATABASE_ROLE=hookwire_app` role switching.
- Updated tests and the HTTP proof script to run API decisions through `hookwire_app` role switching.
- Reordered eligibility checks before `SELECT ... FOR UPDATE`.
- Switched expiry detection to database `now()` in the locked row query.
- Moved required denial reason validation after terminal-state checks so repeat denies safely return `409 already_decided`.
