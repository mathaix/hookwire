## Database Query Proof

The artifact [2026-06-20-issue-009-audit-proof.json](2026-06-20-issue-009-audit-proof.json) is generated by a reproducible script that:

1. Starts a disposable `postgres:16` container.
2. Runs the repository migration runner.
3. Seeds Acme and Globex tenants, Acme users, project membership, and sample audit events.
4. Queries audit data through `hookwire_app` with `app.current_organization_id`.
5. Attempts update and delete against visible audit rows through `hookwire_app` and records permission-denied results.
6. Builds and starts the Next.js app on port `3029`.
7. Captures browser screenshots for timeline, filters, and redacted event detail.

The final proof shows:

- Database rows cover `approval.requested`, `approval.approved`, `policy.changed`, `route.changed`, `key.registered`, `key.revoked`, `session.claimed`, and `local_override.used`.
- Event rows include actor type, actor name, project name, entity type, and redacted metadata.
- `globexDirectRlsProbe` is `[]`, proving the Acme tenant cannot read Globex audit rows through RLS.
- `appendOnly.updateDenied` and `appendOnly.deleteDenied` both return PostgreSQL code `42501`.
- Browser proof shows `[REDACTED]` and no raw secret fixture hits.

## Acceptance Criteria Mapping

- Approval creation and decision events are visible: covered by domain seed tests, DB event rows, E2E timeline assertions, and timeline screenshot.
- Policy changes are visible: covered by service/event tests, DB proof, E2E timeline assertions, and timeline screenshot.
- Local override events can be represented: covered by domain/service tests, E2E detail test, DB proof row, and redacted detail screenshot.
- Audit details never expose unredacted secrets: covered by recursive redaction unit tests, service persistence/read redaction tests, E2E negative assertions, and proof JSON `rawSecretHits: []`.

## Verification Constraint Mapping

- Event creation tests: `tests/unit/audit-timeline.test.ts` creates audit events for approval creation, approval decision, policy change, route change, key registration, key revocation, session claim, and local override.
- Filter tests: unit and E2E tests cover project, session/entity, approval request, policy, and user filters.
- Redaction tests: unit tests cover nested sensitive keys, token-shaped strings, arrays, primitives, and undefined metadata; E2E tests prove raw fixture values are absent from event detail.
- Append-only tests: service and schema tests prove `hookwire_app` cannot update or delete audit rows, while insert/select still work.

Claude specifically confirmed:

- All required event types are modeled, validated, persisted by tests, and visible in E2E.
- Redaction runs both before persistence and again on read, with unit, E2E, and proof coverage for the secret fixtures.
- `audit_events` is covered by forced RLS and tenant-scoped query proof.
- `hookwire_app` is denied update and delete on `audit_events`, proving append-only behavior at the privilege layer.
- User actors require `actorUserId`; non-user actors fall back to typed labels; same-organization membership FKs protect actor attribution.
- Backend filters cover project, entity type plus entity id, and actor user.

Residual risks noted:

- The rendered `/audit` page still uses deterministic client-side seed data; the RLS-backed audit service is implemented and tested but not yet wired into authenticated page data.
- The UI filters by entity type, not a specific entity id; backend filtering supports specific session, approval request, and policy ids.
- Value-pattern redaction is best-effort for freeform strings and should be treated as a fallback behind structured source redaction.
- The app role can still insert arbitrary audit rows, so append attribution is trusted-backend behavior rather than a database guarantee.

Disposition:

- Fixed Claude's proof-script residual by adding hard assertions for required event types, RLS probe emptiness, append-only denial, redaction presence, raw secret absence, and browser console cleanliness. `npm run proof:issue009:audit` passes with those assertions.
- Accepted the seeded UI/service-backed data split as consistent with the current M1 pattern used by sessions, policies, and routes; authenticated service wiring remains a later backend/page integration task.
- Accepted entity-id UI filtering as follow-up because the backend contract and unit tests cover specific session, approval request, policy, project, and user filters required by the issue.
- Accepted value-pattern redaction as a defensive fallback and documented that structured source redaction remains the primary control.
