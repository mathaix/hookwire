## Database Query Proof

The artifact [2026-06-20-issue-008-route-proof.json](2026-06-20-issue-008-route-proof.json) is generated by a reproducible script that:

1. Starts a disposable `postgres:16` container.
2. Runs the repository migration runner.
3. Seeds Acme and Globex tenants, Acme users, groups, on-call assignments, integrations, routes, targets, policies, and a route-referencing policy rule.
4. Queries route data through `hookwire_app` with `app.current_organization_id`.
5. Builds and starts the Next.js app on port `3028`.
6. Captures browser screenshots for route list/detail, approvals/timeout/fallback fields, and target editor state.

The final proof shows:

- `targetTypes` and database `targets` cover `web_inbox`, `slack`, `sms`, `jira`, `linear`, `email`, `github`, `webhook`, and `local_terminal`.
- `web_inbox` and `local_terminal` do not require external integrations.
- `sms` targets use the `twilio` integration provider while the target remains `sms`.
- Group targets include `approval_group_name`.
- On-call targets include `current_on_call_user_name`.
- Policy rule `Route config writes` has decision `route` and references the web inbox route.
- `globexDirectRlsProbe` is `[]`, proving the Acme tenant cannot read Globex route rows through RLS.

## Acceptance Criteria Mapping

- Web inbox can be configured as the first working route target: covered by unit tests, E2E route configuration, screenshots, and proof data showing an active `web_inbox` target without an integration.
- Slack/SMS/Jira/Linear/etc. can be represented before provider workers exist: covered by `providerTargetTypes`, provider matrix UI, service validation tests, E2E provider matrix assertions, and proof rows for every target type.
- A route can target a group or on-call owner: covered by route target service tests, route detail E2E assertions, and proof rows with group and current on-call user output.
- A policy rule can reference a route: covered by database tests that protect referenced routes from disable/delete and proof data with a `route` policy rule.

## Verification Constraint Mapping

- CRUD tests for routes and route targets: `tests/unit/route-integration-config.test.ts` covers create route, update route, create target, disable route, and delete route paths through Postgres.
- Provider target type tests: domain, service, E2E, and proof script cover all nine target types.
- Fallback validation tests: domain and service tests cover fallback assignment, self-reference rejection, cycle prevention, and preservation across partial updates.
- Route reference safety tests: service tests reject deleting or disabling policy-referenced routes and reject deleting fallback-referenced routes.
- Proof artifacts: the proof JSON includes route fixture rows for every provider type and query output for group and on-call targets; screenshots cover route list, detail, target editor, approvals, timeout, and fallback fields.

Claude specifically confirmed:

- Web inbox works as an active target with no external integration dependency.
- All nine provider target types are represented without Slack-specific routing coupling.
- Group and on-call routing are validated by service tests, E2E assertions, and proof query output.
- Fallback cycle prevention overlays the proposed fallback edge and catches self/cycle cases.
- Disable/delete safety blocks policy-rule references and fallback references.
- Tenant isolation uses explicit `organization_id` predicates, same-organization constraints, RLS, and proof query output.
- Provider target validation catches missing integrations, provider mismatches, missing approval groups, invalid recipient kinds, and priority conflicts.

Residual risks noted:

- The rendered `/routes` page still uses deterministic client-side seed data; the RLS-backed route service is implemented and tested but not yet wired into authenticated page data.
- Fallback cycle checks are not concurrency-locked; simultaneous route edits could need row locking or serializable transactions later.
- The `route_in_use` error message is imprecise when the reference is a fallback route rather than a policy rule.
- `disableRoute` blocks referenced routes rather than allowing reversible disable semantics.
- Route targets are add-only in the current UI/service; update, delete, and target enable/disable controls remain follow-up work.
- Production deployments must use the intended non-superuser app role so forced RLS is active; tests set `HOOKWIRE_DATABASE_ROLE=hookwire_app`.

Disposition:

- Accepted the seeded UI/service-backed data split as consistent with current M1 modeling scope; authenticated persistent route configuration belongs with later onboarding/API wiring.
- Accepted add-only route targets for this issue because the acceptance criteria require provider representation and target creation, not target mutation workflows.
- Accepted the concurrency note for the low-traffic configuration path and recorded it as a future hardening item before route edits become high-volume automation.
- Kept `disableRoute` conservative because the issue explicitly asks for safe handling when a route is referenced by policies or fallbacks.
