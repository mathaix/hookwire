## Database Query Proof

The artifact [2026-06-20-issue-006-session-proof.json](2026-06-20-issue-006-session-proof.json) is generated by a reproducible script that:

1. Starts a disposable `postgres:16` container.
2. Runs the repository migration runner.
3. Seeds Claude Code, Codex, OpenClaw, and other-tenant sessions.
4. Queries rollups and detail rows through `hookwire_app` with `app.current_organization_id`.
5. Builds and starts the Next.js app on port `3026`.
6. Captures browser screenshots for filtered list and detail states.
7. Fails if the proof JSON or rendered body contains the known secret fixtures.

The queried rows show:

- Acme sees `claude-a88c`, `codex-7f31`, and `openclaw-19b2`.
- The filtered `codexHigh` query returns only `codex-7f31`.
- The Codex detail query links two `hook_events`, one `approval_requests` row, and one `approval_decisions` row.
- `globexLeakProbe` is `null` when Acme asks for a Globex session by id.
- `globexDirectRlsProbe` is `[]` when Acme queries the Globex session id without an app-level organization filter, proving RLS itself blocks the row.
- The Globex tenant can see only `globex-codex`.

## Acceptance Criteria Mapping

- Users can see Claude, Codex, and OpenClaw sessions in one place: unit tests, e2e tests, screenshots, and query proof cover all three agent types.
- Session detail links approvals and hook events: unit/service tests and e2e detail assertions cover hook events, approval requests, and decisions for `codex-7f31`.
- Sensitive payloads remain redacted: unit tests seed raw secret values into event payloads and assert service output removes them; e2e asserts the browser never renders the secret fixtures; the proof script fails if proof JSON contains those fixtures.
- Sessions can be filtered by project and agent type: unit/service/e2e tests cover project and agent filters, plus status, risk, and date filters required by the verification constraints.
- Tenant isolation: service tests and proof artifacts run under `hookwire_app` and tenant GUC; direct RLS probes verify other-tenant rows are hidden without relying only on app-level filters.

## Notes

- The rendered `/sessions` page uses deterministic seeded fixtures, matching the current web MVP pattern used by the approval inbox. The database-backed service is implemented and tested, but wiring authenticated user context into the rendered page is left for the later onboarding/identity and relay API issues.
- The domain seed and SQL service produce the same `SessionSummary` and `SessionDetail` shapes; keep them in sync until the UI moves fully to the service-backed path.
