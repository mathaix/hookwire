## Database Query Proof

The artifact [2026-06-20-issue-007-policy-proof.json](2026-06-20-issue-007-policy-proof.json) is generated by a reproducible script that:

1. Starts a disposable `postgres:16` container.
2. Runs the repository migration runner.
3. Seeds Acme and Globex tenants, Acme users, routes, policies, and policy rules.
4. Queries policy data through `hookwire_app` with `app.current_organization_id`.
5. Builds and starts the Next.js app on port `3027`.
6. Captures browser screenshots for rule creation, route/override detail, and ordering.

The final proof shows:

- Policy `Deploy guard` at version `5`.
- `Route guarded config writes` at priority `10`, decision `route`, same-tenant `routeId`, `localOverrideAllowed: true`, `requireOverrideReason: true`, and `maxScope: "session"`.
- `Deny destructive deletes` at priority `20`, decision `deny`, no route, no override, and `maxScope: null`.
- `globexDirectRlsProbe` is `[]`, proving the Acme tenant cannot read Globex policy rows through RLS.
- The serialized relay bundle includes deterministic priorities, filtered routes, matcher JSON, route decisions, local override flags, reason requirements, scope, and enabled state.

## Acceptance Criteria Mapping

- Users can create and edit policy rules: covered by domain CRUD tests, service CRUD tests, and the `creates and edits a routed policy rule with override settings` E2E test.
- Rules can point to routes: covered by route decision tests, DB route constraints, the proof query output, and route screenshots.
- Rules can require override reasons: covered by UI controls, domain/service normalization tests, relay bundle serialization, and screenshots.
- Rule order or priority is explicit: covered by deterministic `sortRules`, DB priority uniqueness, unit reorder tests, E2E reorder tests, and DB query proof.
- Policy data can be serialized for local relay cache later: covered by inline serialization snapshot tests and the proof JSON serialized bundle.

## Claude Review

Final review command:

```sh
claude -p "Final blocking-only review for Hookwire issue 007 exact current diff..."
```

Claude emitted a local configuration warning before review:

```text
Permission deny rule "Dont read .env file" matches no known tool — check for typos.
```

Claude review result:

```text
No blocking or high-risk functional/code issues remain.
```

Claude specifically confirmed:

- `localOverrideAllowed=false` clears `requireOverrideReason` and `maxScope` in domain, service, and UI paths.
- DB route and scope constraints match service behavior.
- Reorder uses a transaction-safe negative-then-positive priority rewrite.
- Tenant isolation uses validated role setting, tenant GUC, RLS, and explicit `organization_id` predicates.
- Invalid regex syntax and empty matchers are handled safely for current simulator usage.

Residual risks noted:

- Editing `0001_initial_schema.sql` in place is safe for fresh MVP setups but would need a forward migration if any environment had already applied the original migration.
- The rendered `/policies` page still uses deterministic client-side seed data; the RLS-backed policy service is implemented and tested but not wired into authenticated page data yet.
- User-authored `commandPattern` values can still be expensive regexes; relay-side evaluation should use time bounds or a safer regex engine.
- The simulator button is decorative because evaluation is reactive.
- The builder does not yet expose an enabled/disabled rule toggle.

Disposition:

- Fixed all route/override consistency findings before final review: non-route decisions clear `routeId`, non-overridable rules clear `requireOverrideReason` and `maxScope`, and empty matchers are rejected in domain/service/UI paths.
- Accepted the seeded UI/service-backed data split as consistent with current M1 web MVP altitude; authenticated service wiring is left to later onboarding/identity/API issues.
- Accepted the regex risk for the client-only simulator and recorded that relay-side policy evaluation must add bounded regex execution before consuming untrusted commands at runtime.
